Jobs-to-be-Done: Summary

For this project, I used the jobs-to-be-done framework to understand and design for meeting user needs and goals. There were several of jobs-to-be-done that I create for each user group.

FOR MATERIAL AQUISITION STAFF/STUDENT EMPLOYEES

When an item order from faculty, or a item suggestion from non-faculty is submitted, I want to track the progress of that item through the various stages of the aquisition process, automatically request needed information from librarians, and notify the individual who submitted the request regarding it’s status in the system, so I can more quickly and easily manage requests, attain information from subject librarians regarding requests, and maintain better communication with the requester.

FOR LIBRARIANS

When an item suggestion is assigned to me, I want to be able to view information about the item and quickly approve or deny the request, so I can reduce the amount of time I have to spend responding to emails.

FOR NON-FACULTY PATRONS

When I submit an item purchase suggestion, I want to updated so that I know whether the item I requested has been approved for purchase, and when the item arrives at the library, so I can get access to it as quickly as possible.

FOR FACULTY PATRONS

When I submit an item to be ordered—especially an order that needs to be expedited, I want to be updated so that I know when the item I requested has been ordered, and when the item arrives at the library, so I can use it for research or teaching as quickly as possible.

Wireframing / Prototyping

Once I had created and validated the user flows, I began iterating design solutions.

OWNING A MISTAKE

I didn’t take pictures of my wireframe sketches—all of which I did on a whiteboard.

Since I began both this project—and the Instruction Scheduler project around the same time, I made this same mistake for both projects.

OWNING ANOTHER MISTAKE

I skipped doing wireframes and went straight to high fidelity mockups.

That means I don’t have wireframes to show for this project, but I do have a few screen shots from the initial iterations that I tested. (See first screen of the high fidelity mockup section)
